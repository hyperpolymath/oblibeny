This document is a newcomer companion.
It is not a README and contains no installation instructions.
Its purpose is to explain — plainly and deeply — what Oblíbený is, why it was designed the way it was, and why those design choices matter.

If you want to get started quickly, go to `README.adoc`.
If you want the formal language specification, go to `SPEC.core.scm` or `docs/LANGUAGE-SPEC.md`.
This document is for understanding the shape of the project before you touch any of that.

== What Oblíbený Is

Oblíbený is a dual-form programming language for secure edge computing.

"Dual-form" means it is, in a precise technical sense, two languages in one:

Factory form:: Turing-complete, runs at compile time.
Used to write metaprograms that *generate* constrained-form programs.
The full expressive power of a modern language is available here — recursion,
unbounded iteration, higher-order functions, dependent types (via the AffineScript
host), and arbitrary computation.

Constrained form:: Turing-incomplete, runs at deployment time on the edge device.
This is the form that executes on hardware security modules, smart cards, secure enclaves
(Intel SGX, Arm TrustZone), and IoT edge nodes.
It has no general recursion, no unbounded loops, provably terminates on every input,
has fully reversible core operations, and produces an unmodifiable audit trail.

The constrained form is the thing that matters for security.
The factory form is the thing that makes it practical to write programs in the constrained form.

Only the constrained form is ever shipped to an edge device.
The factory form — and everything generated by it — stays on the build machine.

== Why Turing-Incompleteness Matters

Turing-completeness is a strength in a general-purpose language and a liability in a security substrate.

A Turing-complete language can express the halting problem.
That means no static analysis tool — no matter how sophisticated — can always determine whether a
program will terminate, how much memory it will use, or what operations it will perform.
You can compute approximate answers, but not guaranteed ones.

For the environments Oblíbený targets, approximate is not good enough:

Hardware security modules (HSMs):: An HSM has fixed RAM.
A program that may or may not terminate is a denial-of-service vector.
A program whose memory use cannot be bounded statically is rejected by any serious HSM deployment policy.

Smart cards:: Smart cards have kilobytes of working memory and strict power envelopes.
An unbounded loop that runs for an extra hundred milliseconds can drain a contactless card's
field energy and corrupt a transaction mid-flight.

Secure enclaves (SGX/TrustZone):: The enclave's trusted computing base (TCB) should be auditable.
A Turing-complete program inside a TCB cannot be fully audited — you would need to solve the halting
problem.

IoT edge nodes:: In a deployed fleet of ten thousand sensors, you cannot afford a program that
occasionally loops forever on a particular sensor reading.
You need static guarantees, not probabilistic ones.

The constrained form of Oblíbený gives you those guarantees:

- Every loop has a static bound.
- The call graph is acyclic (no direct or mutual recursion).
- Every for-range upper bound is a compile-time constant.
- `while` and `loop` constructs do not exist at the parser level.
- The static analyzer can compute an upper bound on memory use without running the program.

These are not soft guidelines.
They are enforced by the parser and the constrained checker — a program that violates any of them
does not compile.

=== The "No Rhino" Principle

Think of the constrained form as a door with a specific-sized opening.

The opening is intentionally small — too small for anything dangerous to fit through.
You cannot bring a rhino through a cat door.
The constrained form's entrance is sized to exclude unbounded computation, unauditable control flow,
and unchecked resource use.

The factory form's job is to produce programs that *fit through the door*.
The factory form itself can be as powerful as it needs to be, because it never goes near the door —
it only produces things that do.

Oblíbený is not trying to be a general-purpose language.
It is trying to be a language whose guarantees you can formally verify.
Expressivity is sacrificed deliberately and locally, not accidentally and everywhere.

== The Dual-Form Insight

The dual-form design solves a genuine tension: the constrained form is by construction
hard to write in directly (it lacks recursion, unbounded loops, most of what makes
programming comfortable).
But you need to be able to write real programs in it.

The factory form resolves this.
You write a factory-form program — which can use all of Turing-complete computation — and it
generates a constrained-form program as output.
The constrained-form program is the artifact; the factory-form program is the generator.

This is metaprogramming in the type-safe sense:

- The factory form is run by the Oblíbený compiler, not by the edge device.
- Its output is a constrained-form program that the compiler then validates against the constrained
  form's rules.
- If the generated program violates any constrained-form contract, compilation fails.

The developer's feedback loop is: write a factory-form generator, compile, see whether the generated
program satisfies the constrained-form checker.
This is the same loop as writing a C macro that must produce valid C — but with formal guarantees
instead of convention.

You get the expressivity of Turing-complete metaprogramming and the safety guarantees of a
Turing-incomplete target language.
Neither side compromises the other.

== Reversibility and Accountability

Every runtime operation in the constrained form has a well-defined inverse.

This is not a convenience feature.
It is a core design goal, built into the language from the first operation.

The reversible operation set includes:

- `incr` / `decr` — increment and decrement are each other's inverse
- `push` / `pop` — stack operations are each other's inverse
- `swap` — self-inverse (swap twice returns to original state)
- `xor` — self-inverse (xor twice with same value is identity)

Every operation emits an entry into a `Trace` — an append-only, unmodifiable log of what happened.
The trace is not optional and cannot be suppressed.
It is part of the constrained form's type system.

`Trace` is a non-copyable type.
You cannot duplicate a trace, only extend it.
Once an entry is in the trace, it stays there.

=== The Reversible Debugger

Because every operation has a defined inverse, Oblíbený supports stepping backward through execution.
The reversible debugger reconstructs prior states by applying inverse operations in reverse trace order.

This is not a simulation or an approximation.
It is mathematically exact: if you ran `incr x; incr y`, stepping backward applies `decr y; decr x`
and you are back to the state before either increment.

For security applications, this means:

- Post-incident forensics can reconstruct exactly what happened and in what order.
- Auditors can replay a trace forward and backward to verify that reported state transitions match
  the trace entries.
- The `assert_invariant` primitive records its failure in the trace before halting — you always know
  *which* invariant failed and *at what point*.

Reversibility and accountability are not layered on top of Oblíbený — they are what Oblíbený is for.

== The Echo Type

Some computations are inherently lossy.
A hash function maps many inputs to one output; a projection discards some fields; an aggregation
discards individual values.

In a pure reversibility model, lossy operations are simply forbidden.
That is too restrictive.

Oblíbený introduces the *echo type* to handle lossiness without abandoning accountability.

`echo[A, B]` is the type of a *lossy collapse* from source domain `A` to visible domain `B`.
It is not a product type (it does not say both `A` and `B` are present independently).
It is a *structured residue* of a collapse: the computation lost information going from `A` to `B`,
but `echo[A, B]` retains both the surviving observation and the source constraint that produced it.

The two projections are:

`echo_visible(e: echo[A,B]) : B`:: Returns the surviving observation — the result of the lossy
operation. This is what the outside world sees.

`echo_witness(e: echo[A,B]) : A`:: Returns the retained source constraint — the value that was
present before the collapse. This is what the audit trail retains.

A hash scenario: if `h : A -> B` is a hash function, `echo(input, h(input)) : echo[A, B]` retains
both the hash output (visible) and the original input (witness).
`echo_visible` returns the hash; `echo_witness` returns the pre-image.
From the outside, only the hash is observable; internally, the witness is still there.

=== The Affinity Rule

`echo[A, B]` is affine if and only if `A` or `B` is non-copyable.

Echo does not make copyable data non-copyable by magic.
If both `A` and `B` are copyable primitives (e.g., `i32`, `bool`), `echo[A,B]` is also copyable.
If either is a non-copyable type (a struct, an array, a `Trace`, a `Ref`), the echo is non-copyable.

This is the right behaviour:

- A copyable echo can be freely projected — both projections may be applied to the same binding.
- A non-copyable echo is consumed on first projection — you get one chance to extract either the
  visible or the witness value, not both.
- A non-copyable echo used inside a loop body is rejected — it would be consumed on every iteration,
  which violates affinity.

The affinity rule preserves the copyability discipline of the contents.
It does not add new constraints; it propagates the constraints that already exist.

=== Echo is Not Reversibility

Echo does not participate in `incr`/`decr`/`swap` balancing.
An echo value is not reversible in the operation-inverse sense.
Echo is about *structured lossiness*, not about *reversible computation*.
These are related concerns (both are about accountability) but they are distinct mechanisms.

== Relationship to the Echo-Types Ecosystem

`echo[A, B]` in Oblíbený realises, in a non-dependent constrained language, the echo-types fibre
from the `hyperpolymath/echo-types` Agda formalisation:

----
Echo f y := Σ (x : A), f x ≡ y
----

In the Agda formulation: given a function `f : A -> B` and a specific output `y : B`, an echo
is a dependent pair of a source value `x : A` and a proof that `f x ≡ y`.

Oblíbený operates in a non-dependent type system (the constrained form does not support dependent
types), so the proof term `f x ≡ y` is not carried.
The realisation is the finite-domain executable reading from `hyperpolymath/EchoTypes.jl` (Julia):
a pair `(witness, visible)` that captures the source and the image without a static proof of their
relationship.

The properties that hold across this boundary are:

Non-injectivity is demonstrable:: Distinct witnesses can have equal visible projections at runtime.
The "loss is not total" property:: `echo_witness` recovers the source value even when `echo_visible`
collapses it.
Affinity discipline:: Echo preserves the copyability of its contents (not derived from the Agda
fibre, but consistent with the Julia reading).

What Oblíbený does *not* carry:

- The identity type `f x ≡ y` — no dependent types in the constrained form.
- The functorial echo machinery (`map-over`, `degrade`, `degrade-compose`) — out of scope for the
  constrained form.
- A generic Σ-type — `echo[A,B]` is specific to the accountability use case.

See `docs/echo-alignment.md` for the detailed alignment tracking between Oblíbený and both upstream
projects, and `docs/specs/echo-6a2.adoc` for the formal interface contract.

== The Security Philosophy in Full

The "no rhino" principle extends beyond loops and recursion.

The constrained form is intentionally a *small, analysable, hostile-to-expressivity* language.
Hostility to expressivity is a feature, not a failure of design.
Each constraint that is removed from the constrained form is a constraint that the formal verifier no
longer needs to reason about.

The formal guarantees of the constrained form are:

1. *Termination* — every program terminates on every input.
2. *Bounded resources* — memory use is statically computable.
3. *Full audit trail* — every operation is recorded in an append-only trace.
4. *Reversibility* — every core operation has a defined inverse.
5. *Structured lossiness* — lossy operations are typed and their residues are retained.
6. *Acyclic control flow* — no recursion, no hidden cycles through echo expressions.

These guarantees hold simultaneously and unconditionally.
They are not best-effort or dependent on the programmer not making certain mistakes.
They are enforced by the type system and the constrained checker before a program ever runs.

A formal verifier (such as a Hypatia scan or a gitbot) can mechanically check each of these
properties against a constrained-form program without executing it.
See `docs/CONTRACTILES.adoc` for the full behavioral contracts.

== Who This Is For

Oblíbený is not a general-purpose language and does not pretend to be.
It is a tool for a specific class of problems:

Edge and IoT with accountability requirements::
When you need to know exactly what your deployed code did, in what order, and you need that
knowledge to be verifiable after the fact.

Hardware security modules and smart cards::
When the execution environment has fixed resources, no operating system, and zero tolerance for
non-termination.

Secure enclaves (SGX, TrustZone)::
When the trusted computing base must be auditable and the program inside the enclave must be
formally bounded.

Safety-critical systems::
When termination must be provable by static analysis, not assumed.

Verifiable audit trails::
When the audit log is a first-class part of the system's correctness guarantee, not an afterthought
added by a logging library.

Formal verification projects::
When the program's properties need to be checked by a theorem prover or a static analyzer and
Turing-completeness makes that intractable.

If your problem is not in this list, Oblíbený is probably not the right tool.
Its constraints are only virtues if the problem requires them.
